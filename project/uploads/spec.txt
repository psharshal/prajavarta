2. Design Philosophy and Non-Negotiables


2.1 Core Philosophy

Prajavarta should feel like a modern editorial publishing product, not a regional news portal. Every design decision should be evaluated against this filter.

The reader experience should prioritize, in this order:


Readability of the article


Trust signals and editorial credibility


Content discoverability across the site


Controlled monetization that supports rather than interrupts reading


Speed of perceived load and interaction


2.2 Mobile-First Mandate

Mobile is the primary canvas. Every screen must be designed mobile-first, then expanded to tablet and desktop. Desktop layouts evolve from mobile logic, not the reverse.

The majority of Marathi news consumption happens on mobile devices, often on entry-level Android handsets with mid-tier connectivity. Designing desktop-first and shrinking down produces cluttered mobile screens. Designing mobile-first and expanding up produces calm, scannable layouts at every size.


2.3 The Twelve Non-Negotiables

These rules apply to every screen, every component, every state. There are no exceptions.


Every ad placeholder must exist in Figma with reserved dimensions before frontend handoff.


No ad placement may sit between an H1 headline and the first paragraph of body text.


All interactive elements must have a minimum touch target of 44 by 44 pixels.


All text must meet WCAG 2.1 AA contrast ratios (4.5:1 for body, 3:1 for large text).


Marathi body text must be a minimum of 17px on mobile and 18px on desktop.


Line-height for Marathi body text must be 1.7 minimum.


Article reading column must be capped at 680px on desktop to maintain readability.


Spacing between paragraphs must be 16px minimum.


All cards must follow the unified spacing scale (4, 8, 12, 16, 24, 32, 48, 64).


Sticky elements are limited to one per viewport (header OR sidebar OR anchor ad, not multiple).


Hero images must be a minimum of 1200px wide for Discover eligibility.


Every component must have hover, active, focus, disabled, and loading states defined.



3. Ad Slot Naming Convention

Read this section before any other section. Every page template, every Figma placeholder, every component reference in this document uses these slot codes. Without understanding the naming convention, the page template tables will not make sense.


3.1 Why This Naming Exists

Three teams (Design, Tech, Ad Ops) need to talk about the same ad slot without confusion. A slot called &quot;the second ad on the article page on mobile&quot; is ambiguous. A slot called &quot;A2&quot; is not.

This naming system is mirrored across:


Figma layer names


Frontend component names


Google Ad Manager (GAM) ad unit names


QA references and bug reports


3.2 The Code Structure

Every slot has a two-part code: a letter prefix indicating page type and device, followed by a number indicating position on that page.

3.3 How To Read a Slot Code

Examples to make this concrete:


H2: Homepage, Mobile, second ad position from the top


DH1: Desktop Homepage, first ad position (top billboard)


A2: Article page, Mobile, second ad position


DA2: Desktop Article page, second ad position (sticky sidebar)


C1: Category page, Mobile, first ad position


DC2: Desktop Category page, second ad position (sticky sidebar)


3.4 Number = Position Order, Not Importance

The number in each slot code reflects the order the ad appears as the user scrolls down the page, starting from the top. A1 is the first ad encountered. A5 is the fifth. The numbers are not a ranking of revenue value or priority.


3.5 Mobile and Desktop Are Separate Slots

H1 and DH1 are not the same slot rendered at different sizes. They are independent ad units with different positions, sizes, and behaviors. Mobile users see H slots. Desktop users see DH slots. They never see both.

This separation gives Tech and Ad Ops teams the ability to run different ad strategies per device without conflicts.

The complete list of all 25 slots with sizes, positions, and sticky behavior is documented in Section 6.1 (Master Ad Slot Reference Table). The page template specifications in Section 5 reference these slots by code.

5.4 Tag and Topic Cluster Pages

Tag pages consolidate semantic relationships for Google. They should not be thin archive pages. Each tag page must have visible editorial value.


5.4.1 Required Elements


Tag name as H1 with intro copy (30-100 words explaining the topic)


Latest stories tagged with this entity (10 stories)


Most read stories under this tag (5 stories)


Related tags (semantic suggestions)


Ad slots same as category pages (C1, C2, C3 on mobile; DC1, DC2, DC3 on desktop)


5.4.2 Layout

Tag pages reuse the Category page layout (Section 5.3). The structural difference is content depth: tag pages typically have less editorial curation than category pages, so subcategory modules and evergreen sections may be omitted.


5.4.3 Thin Tag Protection

Tags with fewer than 5 articles must be designed for graceful handling. The empty state must be visually different from a populated tag and must include a noindex tag (Tech responsibility) and contextual recommendations to keep the user engaged.

5.5.2 Layout

Author pages reuse the Category page layout. The Category Header block is replaced with an Author Profile block (photo, name, bio, social links). Ad slots follow the C/DC scheme.

6.2 Ad Density Rules by Article Length

Article ad density is tied to word count. This protects CWV, ad policy compliance, and reading experience.


Article Length
Allowed Slots
Reasoning
Under 300 words
A1, A5 only
Short articles cannot accommodate inline ads without harming UX.
300-600 words
A1, A2, A4, A5
Two inline placements fit comfortably. A3 skipped.
600-1000 words
A1, A2, A3, A4, A5
Standard ad density.
1000+ words
All slots (A1-A6)
Long-form articles can accommodate vignette transitions.




6.2.1 Hard Density Caps


Minimum gap between two inline ads: 4 paragraphs


Maximum ad density: 1 ad per 200 words of body content


First ad cannot appear before the hero image is visible


No ad may appear between the H1 and the first paragraph


Sticky anchor (A5) is the only sticky element on mobile article pages


6.3 Figma Placeholder Standards


6.3.1 Required Annotations

Every ad placeholder in Figma must visually display:

Full slot name (e.g., &quot;Mobile Article Inline Ad 1&quot;)


Ad size (e.g., &quot;300 x 250&quot;)


Sticky behavior (Yes/No)


Device visibility (Mobile/Desktop)


Reserved container height


6.3.2 Layer Naming Convention in Figma


Format: ad-slot-[SlotID]-[size]


Example: ad-slot-A2-300x250


Example: ad-slot-DH1-970x250


6.3.3 Advertisement Label

Every ad container must include an &quot;Advertisement&quot; label above the ad area. This label is part of the reserved height.


Label text: &quot;जाहिरात&quot; (Marathi for Advertisement) or &quot;Advertisement&quot;


Label style: caption token, text-tertiary, uppercase


Label height: 20px


Total reserved height: ad height + 20px label

7.1 Loading States

Loading states use skeleton components, not spinners. Skeletons must match the layout structure they&apos;re replacing.


7.1.1 Required Skeleton Components


Hero card skeleton (image block + 2 text bars)


Standard card skeleton (image block + 2 text bars)


Article body skeleton (4 text bars + image block + 4 more text bars)

Author byline skeleton (circle + 2 text bars)


7.1.2 Skeleton Animation


Type: Subtle shimmer (left-to-right gradient sweep)


Duration: 1.5 seconds per cycle


Color: surface-tertiary base, surface-secondary highlight

8. Accessibility Standards

All designs must meet WCAG 2.1 Level AA standards as a minimum.


8.1 Color Contrast


Element
Minimum Ratio
Notes
Body text on background
4.5:1
Body tokens on surface colors
Large text on background
3:1
Display and heading tokens (24px+ at 700 weight)
UI component borders
3:1
Buttons, form inputs, focus indicators
Iconography
3:1
Functional icons (decorative exempt)




8.2 Focus States

Every interactive element must have a visible focus state designed in Figma.


Width: 2px


Color: brand-primary


Offset from element: 2px


Border-radius: Match the element&apos;s border-radius + 2px


8.3 Touch Targets

Minimum tap target size on mobile is 44x44 pixels for buttons, links, icons, and any interactive element.


8.4 Alt Text Standards


Hero images: Required descriptive alt text (50-150 characters)


Inline images: Required descriptive alt text


Author photos: Required (typically the author&apos;s name)


Decorative icons: alt=&quot;&quot; (empty, marked decorative)


Functional icons: Required descriptive alt text


8.5 Reduced Motion

All animations must have a reduced-motion alternative. Skeleton shimmers become static blocks. Auto-scrolling tickers pause. Carousel auto-advance disables.


8.6 Text Resizing

All text must remain readable when the user zooms to 200%. No fixed-height containers that clip text. No fixed font sizes that ignore user preferences.


8.7 Dark Mode

Dark mode is not in scope for v1 launch but tokens must be designed to support a future dark mode without rework. Use semantic color tokens (text-primary, surface-primary), not hex values directly in components.



9. Article Page Variants

Articles vary significantly. Each variant requires its own Figma frame.


9.1 Standard Text Article

No A1 ad slot if video is the hero


Use for: interviews, press conferences, video reports

Timestamp per update


Auto-refresh every 60 seconds

Reading progress bar at top


Estimated read time prominent in byline


Pull quotes between sections


Section navigation (jump to section)


Author bio expanded with credentials


Use for: investigations, deep analysis, features over 1500 words


9.6 Sponsored/Branded Content


Visible sponsored label at top


Sponsor brand name visible


Visually distinct from editorial (surface-secondary background)


Disclosure block above the body


Use for: branded content partnerships, paid features

The largest, most prominent visual element on a page.

Interaction to Next Paint. Core Web Vital measuring responsiveness.

Google ad format showing multiple ads in a grid.

Revenue Per Mille (per thousand impressions).

Placeholder UI that matches the layout of content while it loads.
