06 Area Logic - Managed as Tags

Areas like Hinjawadi, Baner, Wakad, Kothrud are not separate categories and not separate schema fields. They are managed as geo-tags alongside topic tags on the article.

Why Tags, not a separate field

  • Same tagging interface editors already use
  • Automatically powers a /tag/hinjawadi page if article volume justifies it
  • Feeds the recommendation engine without schema complexity
  • No new routes, no new CMS fields, no new UI

Example article tagging

Category: Pune
Topic: Crime
Tags: #हिंजवडी, #पुणे पोलीस, #सायबर गुन्हे

Recommendations this article feeds

  • Pune Crime stories (city + topic match)
  • Other Hinjawadi tag stories (area match)
  • Maharashtra Crime stories (category match)
  • Pune Police tag stories (tag overlap)
Area tag creation rule: Create the area tag only when it will have 5+ articles per month consistently. Do not create tags for one-off event locations.

10 Automated Content Intelligence

Editors write. The system tags, scores, and distributes — automatically. No manual categorisation required at publish time.

11 Geo-Personalization

The same homepage is served to everyone, but the article scoring and section order shift based on where the reader is. No separate pages are built. No login required. It works on the first visit.

How location is detected on your own server

MaxMind GeoLite2 — a free database file that lives on the server. One local lookup per new visitor. No external API call. No per-request cost. Returns country, state, city within 1ms. Updated monthly. This is the standard used by major news sites worldwide.

For area-level precision (Dadar, Hinjawadi, Baner) — the browser geolocation API is used. It asks the user once for permission. Once granted, area is stored in a cookie and never asked again.

13 The Correct Mental Model: Attributes, Not Folders

Most people imagine news content works like a filing cabinet. One article goes into one folder. That model breaks the moment one article spans two cities, two topics, or both national and local impact.

Old model (breaks at scale)
India
  └── Maharashtra
       └── Politics
           └── article.html

Problem: where does "Pune metro affects Maharashtra economy" go?
City? Transport? Economy? Maharashtra? All of the above.

The correct model treats every article as a set of attributes. The system then asks: where should this article appear? Not: which one folder does it belong to?

Modern model (what Prajavarta is building)

article = {
  locations: [Maharashtra, Pune, India],
  topics: [Politics, Transport],
  entities: [CM, Metro Rail Board],
  importance_score: 87,
  freshness_score: 92,
  location_score: 80
}

→ Maharashtra feed: this article ranks 3rd
→ Politics feed: this article ranks 7th
→ Pune feed: this article ranks 1st
→ Homepage: this article ranks 5th
→ Transport tag: this article ranks 1st

The same article appears in Maharashtra, Politics, Pune, and the homepage simultaneously — at different positions — because each feed computes its own ranking independently. This is exactly how Google News, Dailyhunt, and Inshorts work.

The Netflix analogy: One movie appears under Action, Trending, Indian, New Releases, and Because You Watched XYZ at the same time. Same content. Different ranking contexts. News works identically.

15 Dynamic Feed Engine

Feeds are never generated fresh from the database on every page load. Instead the Feed Engine computes and caches ranked article lists every 15 minutes. Page renders read from cache. The database is only touched when cache is stale or an editorial override invalidates it.

Weighted Scoring Formula per Feed Context

The weights shift depending on which feed is being computed. A city page cares more about location relevance. A category page cares more about topic relevance. The homepage balances everything.

Redis Feed Caching: Precomputed Feed Blocks

The Feed Engine maintains Redis sorted sets for every feed context. Score is the sort key. Article ID is the value. A page render reads from Redis in under 1ms — no PostgreSQL query for feed ordering.

Redis sorted sets maintained:

feed:category:rajkaran (politics articles, category weights)
feed:category:gurhegeari (crime articles, category weights)
feed:city:pune (Pune articles, city weights)
feed:city:mumbai (Mumbai articles, city weights)

feed:segment:E (outside-India users, जग then देश boosted)

Cache TTL: 15 minutes (cron refreshes before expiry)
Cache bust: immediate when moderator pins or boosts an article
PostgreSQL is the source of truth. Elasticsearch is the discovery and search layer. Redis is the serving layer. Each has a specific role — they are not alternatives. A page request reads from Redis, never directly from PostgreSQL or Elasticsearch.

16 Backend Processing Pipeline

Every article goes through a sequential processing pipeline. Most steps run asynchronously in a background queue (BullMQ). The editor sees publish confirmation immediately — enrichment happens behind the scenes.

Step 1 — Editor saves article in CMS
    Article stored in PostgreSQL: status = "draft" or "pending_review"
    BullMQ job queued immediately

Step 2 — Keyword Scanner (background, <100ms)
    Scans headline + first 3 paragraphs
    High confidence → city, topic, area tags applied directly
    Low confidence → flags for AI tagger

Step 3 — AI Tagger (only when needed, 2-3 sec)
    Article text sent to Claude API
    Returns: category, city, area[], entities[], tags[] with confidence scores
    Cost per article: under Rs. 0.25

Step 4 — Metadata Extraction
    article_category_map rows created (one per category, with relevance_score)
    article_locations rows created (location hierarchy resolved automatically)
    Named entities linked to entities table

Step 5 — Moderator Review (if submitted by Reporter)
    CMS shows article with suggested tags pre-filled
    Moderator can correct any tag before approval
    Moderator approves → status = "approved"

Step 6 — Publishing
    Article status → "published"
    Elasticsearch index updated (full-text and tag indexes)
    Affected Redis feed caches invalidated immediately
    SEO sitemap entry added

Step 7 — Score Recalculation (cron, every 15 min)
    final_score computed for all published articles (last 7 days)
    Redis sorted sets updated per feed context
    Trending list updated: top 5 by pageviews_last_2hrs

Step 8 — Reader Page Request
    API reads from Redis → returns ranked article list (<1ms)
    PostgreSQL only queried for article body content (by ID, indexed)

Why Async Queue Matters

AI tagging takes 2-3 seconds. Elasticsearch indexing takes 300-500ms. If these ran synchronously, every editor save would block for 3-4 seconds. The BullMQ queue decouples saving from processing. The editor saves, sees confirmation in under 200ms, and the article is enriched in the background.

A Moderator reviewing an article in the CMS always sees the AI-suggested tags pre-filled. Corrections are rare (estimated 5-10% of articles). This is the human-in-the-loop checkpoint before any article goes live.

Final Section — Read This First
How Everything Works Together
The complete system in plain language. Covers Sections 00-18. Readable standalone without reading the full document.
When an Editor Publishes an Article
The editor writes the article and hits publish.

The moment it is saved, a keyword scanner reads the headline and first few paragraphs. It identifies the city (पुणे, मुंबई), the topic (Crime, Politics, Sports), and any area names (Hinjawadi, Dadar). Tags are applied in under a second.

If the scanner is unsure — say the article spans two cities or two topics — it sends the text to an AI which returns precise tags. This takes 2–3 seconds and costs less than a rupee.

The article is now fully tagged. No editor action needed for categorisation.

Every 15 minutes, a background process recalculates scores for all recent articles. Breaking news gets a 100-point boost. Articles getting heavy traffic get a velocity boost. Older articles decay. The result is a ranked list that powers the homepage hero, secondary slots, and trending module.

If an editor wants to force any article to the hero position — they set the Main Hero label in the CMS. The algorithm respects it immediately.
When a Reader Opens the Site
The server receives the request and reads the reader's IP address. A local database lookup (under 1ms) returns the reader's country, state, and city. This happens before the page starts rendering.

A reader from Pune gets a Maharashtra hero and a Pune story in the secondary slot. A reader from Delhi gets a National (देश) hero with Maharashtra in the secondary. A reader from Dubai gets Maharashtra in the hero — they are likely Marathi diaspora.

This location is stored in a cookie. Every visit after the first is instant — no lookup needed.

If the reader grants location permission in the browser, the system also detects the area (Dadar, Hinjawadi, Kothrud). A small strip of hyper-local area news appears if 3 or more stories from that area exist.

While reading an article, one related story appears mid-article — always from the same city and same topic. At the end of the article, 6 related stories appear — first 3 from the same city and topic, next 2 from the same topic in other cities, and one trending pick as a discovery hook.

Trending is never personalised. What Maharashtra is talking about right now is the same for everyone.
The Complete Flow — Article to Reader
Reporter writes article → saved in PostgreSQL (status: draft)
    → BullMQ job queued immediately (async, editor sees save confirmation in <200ms)

Background queue processes the article
    → Keyword scanner: identifies city, topic, area, entities (<100ms)
    → High confidence → tags applied directly to article_category_map
    → Low confidence → Claude API called → structured tags + confidence scores returned
    → Location hierarchy resolved: Pune → Maharashtra → India (via parent_id)

Reporter submits for review
    → Moderator opens article in CMS — sees AI-suggested tags pre-filled
    → Moderator corrects if needed (rare), approves → status: published
    → Elasticsearch index updated (full-text + inverted indexes)
    → Affected Redis feed caches invalidated immediately

Every 15 minutes (background cron)
    → Recalculate final_score: freshness(30%) + location(20%) + topic(15%) + velocity(15%) + editorial(10%) + CTR(10%)
    → Breaking flag adds 100 regardless of weights
    → Redis sorted sets updated per feed: homepage, category, city, segment
    → Trending cache updated: top 5 by pageviews in last 2 hours

Reader opens prajavarta.com
    → IP lookup: MaxMind GeoLite2 returns country + state + city (1ms, own server)
    → Segment assigned: A (MH+Area) / B (MH+City) / C (MH) / D (Outside MH) / E (Outside India)
    → Feed API reads from Redis sorted set for that segment (<1ms)
    → Homepage renders with personalised hero, secondary, category order
    → Geo segment stored in cookie pv_geo (30 days, no personal data)

Reader opens an article
    → Inline related: PostgreSQL query, same city + same topic, ranked by score (1 block)
    → End related: 3 same city/topic + 2 other city/same topic + 1 trending (6 cards)
    → Trending module: Redis read, site-wide top 5, same for everyone

Next visit
    → Cookie read instantly → personalisation applied → no GeoLite2 lookup needed

Delete or retract a published article (Super Admin only)
Phase 1 (Foundation) requires one backend developer for 6-8 weeks: PostgreSQL schema with article_category_map and location hierarchy, Redis feed caching, BullMQ queue, keyword auto-tagger, cron scoring engine, RBAC, and GeoLite2 geo-detection. No machine learning team. No data science. Standard database queries plus one keyword dictionary. Phase 2 adds the Claude API tagger (2-3 days extra) and Elasticsearch (2-3 weeks). Phase 3 adds user accounts and personalised feeds.
Dev implementation note (Phase 1):
Tables: articles, article_category_map, categories, locations, article_locations, tags, entities, article_scores, users, roles.
Article fields: title, slug, content, status, author_id, language, published_at, breaking_flag, editorial_label, pin_to_homepage, boost_score, expire_boost_at.
Score fields (article_scores): final_score, velocity_score, freshness_score, ctr_score, editorial_score.
Routes: /city/[slug] alongside existing /category/[slug]. Area tags route: /tag/[slug].
Services: BullMQ worker for async tagging pipeline. Cron every 15 min for scoring + Redis refresh. MaxMind GeoLite2-City.mmdb for geo-detection (free, self-hosted, monthly update).
Stack: NestJS + PostgreSQL + Prisma + Redis + BullMQ. Elasticsearch added in Phase 2. Claude API tagger optional, added after keyword dictionary is tuned on real articles.
